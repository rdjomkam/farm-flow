# BUG-042 — Impossible de scroller dans le dialog "Nouvelle vague" sur mobile
**Sévérité :** Haute
**Détecté par :** QA (session Playwright mobile 2026-04-23) — rapporté par @pre-analyst
**Sprint :** Bugfix (post Sprint 11/12)
**Fichier(s) :**
- `src/components/ui/dialog.tsx` (wrapper partagé — cause racine)
- `src/components/vagues/vagues-list-client.tsx` (dialog concret observé)
- `src/app/globals.css` (safe-area utilitaires)
- `src/app/layout.tsx` (viewport meta `viewport-fit=cover` — déjà OK)
- ~51 autres dialogs qui n'utilisent PAS `<DialogBody>` (voir pre-analysis)

## Description
Sur mobile (viewport 375×812, preset Playwright iPhone), depuis `/vagues`, ouvrir le dialog
"+ Nouvelle vague" (New batch) affiche un contenu d'environ 1100 px de haut (4 sections :
Identification, Population initiale, Configuration d'élevage, Bacs — puis `[Annuler][Créer la vague]`).

Comportement observé :
- Le bouton de soumission est positionné à `top: 1053px`, hors de la viewport 812 px.
- Le `DialogContent` **ne scrolle pas** : ni le viewport de la page, ni aucun ancêtre interne.
- `window.scrollTo(...)` est sans effet (la page de fond est `overflow:hidden` quand le Dialog est ouvert).
- Aucun élément scrollable n'existe entre `<DialogContent>` et le formulaire.

Comportement attendu :
- Le dialog doit être entièrement parcourable au doigt.
- Le footer `[Annuler][Créer la vague]` doit rester atteignable (idéalement collant / sticky
  en bas) sans masquer la dernière section du formulaire, et sans empiéter sur la zone
  home indicator iOS (`env(safe-area-inset-bottom)`).

Impact utilisateur : le flux de création d'une vague est **impossible à finaliser sur mobile**.
Seule l'automatisation via `form.requestSubmit()` fonctionne, ce qui confirme que la submit
est logiquement correcte — c'est bien un défaut de layout mobile.

## Étapes de reproduction
1. Ouvrir l'application en viewport mobile 375×812 (DevTools → iPhone 12/13, ou Playwright
   `page.setViewportSize({ width: 375, height: 812 })`).
2. Se connecter avec un compte disposant de `VAGUES_CREER`.
3. Naviguer vers `/vagues`.
4. Tapoter le bouton `+ Nouvelle vague` en haut de liste.
5. Le dialog s'ouvre en plein écran mobile (classe `inset-0 rounded-none`).
6. Tenter de faire défiler le contenu du formulaire vers le bas (glissement vertical
   intérieur au dialog) — **aucun défilement**.
7. Le footer avec `[Annuler][Créer la vague]` reste invisible, hors écran.
8. Constat : `document.querySelector('[role="dialog"]').scrollHeight` ≈ 1100,
   `.clientHeight` ≈ 812, mais l'élément n'a **pas** `overflow-y:auto`.

## Cause racine
Le wrapper partagé `DialogContent` (`src/components/ui/dialog.tsx` lignes 62-67) construit
un conteneur interne `flex flex-col h-full md:max-h-[85dvh] overflow-x-clip`. Sur mobile :
- `h-full` borne la hauteur mais **aucun `overflow-y-auto`** n'est déclaré.
- `md:max-h-[85dvh]` n'a d'effet qu'à partir de `md:` (≥768 px) — absent sur mobile.
- Le mécanisme "scrollable" n'est disponible **que si** le dialog concret utilise
  `<DialogBody>` (qui a `flex-1 overflow-y-auto`). Le dialog "+ Nouvelle vague"
  (`vagues-list-client.tsx` lignes 207-386) **n'utilise pas** `<DialogBody>` :
  `<form>` est un enfant direct de `<DialogContent>`.

En conséquence, quand la somme des sections dépasse la hauteur de la viewport, le contenu
déborde silencieusement sans scroll. ~51 autres dialogs sont dans le même cas (voir
pre-analysis).

À noter : ERR-064 (ERRORS-AND-FIXES.md) a documenté un pattern similaire pour les
Sheets. Le même principe s'applique ici : garantir un layout flex-col en trois zones
(header / body scrollable / footer sticky avec safe-area bottom).

## Fix
- [x] `src/components/ui/dialog.tsx` : patcher le wrapper interne de `DialogContent` pour
      que le conteneur soit **toujours** borné en hauteur ET scrollable par défaut :
      - mobile : `h-[100dvh] max-h-[100dvh] overflow-y-auto` (au lieu de `h-full`)
      - desktop : `md:h-auto md:max-h-[85dvh]`
- [x] `src/components/ui/dialog.tsx` : rendre `DialogFooter` **sticky** par défaut
      (`sticky bottom-0 z-10 bg-card border-t border-border/40 md:border-0` +
      `pb-[max(1rem,env(safe-area-inset-bottom))]` préservé).
- [x] Test de non-régression Vitest ajouté :
      `src/components/ui/__tests__/dialog-scroll.test.tsx` (2 tests, passent).
