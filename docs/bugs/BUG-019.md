# BUG-019 — prisma.activite undefined au runtime (TypeError findMany)

**Severite :** Critique
**Detecte par :** @ronald
**Sprint :** 11
**Fichier(s) :** src/lib/queries/activites.ts:33, src/app/planning/page.tsx:17

## Description

La page `/planning` crash au chargement avec un `TypeError: Cannot read properties of undefined (reading 'findMany')`. Le probleme est que `prisma.activite` est `undefined` au runtime. Meme symptome que BUG-017 et BUG-018 — meme cause racine (client Prisma non regenere).

## Etapes de reproduction

1. Lancer l'app en dev (`npm run dev`)
2. Naviguer vers `/planning`
3. La page crash avec le TypeError

## Cause racine

**Identique a BUG-017 et BUG-018 : client Prisma non regenere apres la migration Sprint 11 (`20260311100000_add_alertes_planning`).**

Le modele `Activite` a ete ajoute dans la migration `add_alertes_planning` avec les tables `ConfigAlerte`, `Notification`, `Activite`. Le client genere `src/generated/prisma/` n'avait pas ete regenere apres cette migration, donc la propriete `prisma.activite` n'existait pas sur le singleton `PrismaClient` en memoire.

**Verification effectuee :**
- Table `Activite` presente en DB : OUI (confirmee via `\dt`)
- Migration `20260311100000_add_alertes_planning` appliquee : OUI (finished_at renseigne)
- Fichier `src/generated/prisma/models/Activite.ts` existait : OUI (mais client pas regenere)
- Getter `get activite()` dans class.ts apres `prisma generate` : ligne 429

## Fix applique

Meme fix que BUG-017 et BUG-018 — les trois bugs partagent la meme cause racine et ont ete corriges ensemble :

### 1. Regeneration du client
```bash
npx prisma generate
```

Le getter expose dans `src/generated/prisma/internal/class.ts` (ligne 429) :
```typescript
get activite(): Prisma.ActiviteDelegate<ExtArgs, { omit: OmitOpts }>;
```

### 2. Nettoyage cache
```bash
rm -rf .next
```

### 3. Prevention dans `package.json`
```json
"build": "prisma generate && next build",
"postinstall": "prisma generate"
```
