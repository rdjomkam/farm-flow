# BUG-023 — Page Tarifs : fetch self-API échoue en SSR → plans vides
**Sévérité :** Critique
**Détecté par :** @project-manager
**Sprint :** 33
**Fichier(s) :** src/app/tarifs/page.tsx

## Description
La page "Plan & tarifs" (`/tarifs`) n'affiche aucun plan. Le Server Component tente de fetch sa propre API
via `fetch(\`${baseUrl}/api/plans?public=true\`)` (ligne 30), mais :

1. **Les variables d'env `NEXTAUTH_URL` et `NEXT_PUBLIC_BASE_URL` ne sont pas définies** dans `.env` → fallback sur `http://localhost:3000`
2. **Un Server Component ne peut pas fiablement fetch sa propre API** pendant le SSR — le serveur n'est pas toujours accessible à cette URL (port différent, pas encore prêt, etc.)
3. **L'erreur est silencieusement avalée** par le `catch` vide (ligne 37-39) → `plans` reste `[]`
4. Le composant `PlansGrid` reçoit un tableau vide et affiche "Les plans ne sont pas disponibles pour le moment."

## Étapes de reproduction
1. Ouvrir `/tarifs`
2. Observer : aucun plan n'est affiché, juste le message "Les plans ne sont pas disponibles"

## Cause racine
Anti-pattern Next.js : un Server Component qui fetch sa propre API route au lieu d'appeler directement
la fonction Prisma/query. Le fetch échoue car le serveur ne peut pas se joindre lui-même de manière fiable pendant le rendu SSR.

## Fix proposé
Remplacer le `fetch()` par un appel direct à `getPlansAbonnements()` depuis `src/lib/queries/plans-abonnements.ts` :

```typescript
// AVANT (broken)
const res = await fetch(`${baseUrl}/api/plans?public=true`, { cache: "no-store" });
const data = await res.json();
plans = data.plans ?? [];

// APRÈS (correct)
import { getPlansAbonnements } from "@/lib/queries/plans-abonnements";
const plans = await getPlansAbonnements(false);
```
