# BUG-041 — Tank count shows "0 tank" after creating a vague with assigned tanks
**Sévérité :** Haute (upgraded from Moyenne after pre-analysis — docs/analysis/pre-analysis-bug-041.md)
**Détecté par :** manual QA (preview test 2026-04-23)
**Sprint :** —
**Fichier(s) :** src/components/vagues/vagues-list-client.tsx (probable — to verify), src/app/api/vagues/route.ts (POST payload), src/components/vagues/new-vague-dialog.tsx (form submission)

## Description
After creating a new vague via the "New batch" dialog with a tank selected (Etang A, 1000 fish, "Distribution balanced."), the new vague appears in the list showing **"0 tank"** in the card meta row, even though the form submission clearly assigned the tank with the correct fish count.

Expected: the card should show **"1 tank"** (or however many tanks were assigned).

## Étapes de reproduction
1. Login as `admin@dkfarm.cm` / `admin123`
2. Select site **Ferme Douala**
3. Navigate to `/vagues`
4. Click **+ New batch**
5. Fill:
   - Batch code: `VAGUE-2026-TEST`
   - Start date: 23.04.2026
   - Number of fry: 1000
   - Initial weight: 5g
   - Farming configuration: Clarias Standard Cameroun
   - Tank: check Etang A → click "Distribute evenly" → shows "Total distributed: 1000 / 1000 fish" + "Distribution balanced."
6. Click **Create batch**
7. Modal closes, list shows new vague as `0 tank`

## Cause racine (à investiguer)
**Confirmed during follow-up mobile retest (2026-04-23)**: the tank IS persisted in the database. Proof: when attempting to create a second vague using the same tank (Etang A), the API correctly rejects it with `409 Conflict — "Bacs déjà assignés à une vague : Etang A"`. So the write path is correct.

**Root cause — confirmed via DB inspection (2026-04-23):**

Two tables track tank-vague relationships:
1. `Bac.vagueId` (direct FK on Bac, legacy)
2. `AssignationBac` (pivot table with lifecycle dates + population — modern tracking)

The POST `/api/vagues` route **only updates `Bac.vagueId`** but does **not create the corresponding `AssignationBac` row**. The list query displaying tank count likely counts `_count.assignationsBac` (new system) → returns 0 → "0 tank" displayed.

DB state after creating VAGUE-2026-TEST with Etang A:
```
Bac.vagueId        → SET to cmobew19k0001mxh8qxuh7voz  (explains 409 Conflict on reuse)
AssignationBac     → 0 rows for this vagueId           (explains "0 tank" in list)
```

Compare with the seed-created VAGUE-2026-01 which likely has both populated (shows "3 tanks" correctly).
