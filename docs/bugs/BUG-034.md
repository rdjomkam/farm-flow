# BUG-034 — LoadingOverlay bloque toute navigation (systémique)
**Sévérité :** Critique
**Détecté par :** @architect
**Sprint :** UX
**Fichier(s) :** src/components/ui/navigation-loader.tsx, src/contexts/global-loading.context.tsx, src/components/ui/loading-overlay.tsx

## Description
`NavigationLoader` (utilisé dans tous les `loading.tsx`) appelle `increment()` sur le `GlobalLoadingContext`, ce qui déclenche `LoadingOverlay` (overlay plein écran z-[9999] avec backdrop blur). Chaque transition de page bloque toute interaction utilisateur. Le `LoadingOverlay` est conçu pour les mutations (soumission formulaire, suppression), pas pour la navigation entre pages.

## Étapes de reproduction
1. Aller sur n'importe quelle page
2. Cliquer sur un lien de navigation (ex: Vagues → Dashboard)
3. Observer l'overlay plein écran qui bloque toute interaction

## Cause racine
Un seul compteur partagé dans `GlobalLoadingContext`. `NavigationLoader` et les mutations utilisent le même `increment()`. `LoadingOverlay` réagit à `isLoading` (le compteur unique) → toute navigation déclenche le blocage plein écran.

## Fix
- Ajouter un second compteur `mutationCount` + `isMutating` dans `GlobalLoadingContext`
- `LoadingOverlay` ne réagit plus qu'à `isMutating`
- `NavigationLoader` continue d'incrémenter le compteur principal (barre fine en haut)
- Les mutations utilisent `incrementMutation()` / `decrementMutation()` pour déclencher l'overlay
- Voir plan détaillé : `docs/decisions/PLAN-ux-filters-loading.md` §P3
