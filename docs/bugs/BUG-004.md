# BUG-004 — Redirect loop infinie sur la page d'accueil (/)
**Severite :** Critique
**Detecte par :** @tester
**Sprint :** 7
**Fichier(s) :** src/middleware.ts, src/app/page.tsx, src/lib/auth/session.ts

## Description
La page d'accueil (/) recharge en boucle non-stop. Le navigateur finit par afficher ERR_TOO_MANY_REDIRECTS.

Ce qui se passe : redirect loop `/` → `/login` → `/` → `/login` → ...
Ce qui devrait se passer : l'utilisateur est redirige vers `/login` une seule fois et voit le formulaire de connexion.

## Etapes de reproduction
1. Se connecter a l'application normalement
2. Attendre que la session expire en DB (ou supprimer la session manuellement en DB)
3. Garder le cookie `session_token` dans le navigateur (il n'est pas efface)
4. Visiter `/`
5. Resultat : la page recharge en boucle indefiniment

Alternative pour reproduire :
1. Se connecter
2. En DB : `DELETE FROM "Session" WHERE "userId" = '<user-id>';`
3. Rafraichir la page → boucle infinie

## Cause racine
Desaccord entre le middleware et les Server Components sur la validite de la session :

### Flux de la boucle
```
[1] GET /
    → Middleware (src/middleware.ts:42-43)
    → Cookie session_token EXISTE → NextResponse.next() (passe)
    → page.tsx:9 → getServerSession() → cherche en DB → session expiree/supprimee → null
    → page.tsx:10 → redirect("/login")

[2] GET /login
    → Middleware (src/middleware.ts:34-36)
    → /login est PUBLIC_ROUTE → verifie cookie → cookie EXISTE
    → NextResponse.redirect(new URL("/", request.url))
    → RETOUR A [1] → BOUCLE INFINIE
```

### Probleme fondamental
- Le **middleware** (Edge Runtime) ne peut PAS acceder a la DB pour valider la session
- Il se base uniquement sur l'EXISTENCE du cookie `session_token`
- Les **Server Components** (Node Runtime) valident la session en DB via `getServerSession()`
- Quand la session est expiree/invalide en DB mais le cookie existe encore → ils se contredisent

### Code fautif — src/middleware.ts lignes 34-36 :
```typescript
const sessionToken = request.cookies.get(SESSION_COOKIE)?.value;
if (sessionToken && PUBLIC_ROUTES.includes(pathname)) {
    return NextResponse.redirect(new URL("/", request.url));  // ← CAUSE DE LA BOUCLE
}
```

## Fix propose

### Option A — Supprimer le redirect automatique /login → / (recommande)
Supprimer les lignes 34-36 du middleware. Ne plus rediriger automatiquement les utilisateurs
qui visitent `/login` ou `/register` quand un cookie existe.

```typescript
// AVANT
if (isPublicRoute(pathname)) {
    const sessionToken = request.cookies.get(SESSION_COOKIE)?.value;
    if (sessionToken && PUBLIC_ROUTES.includes(pathname)) {
        return NextResponse.redirect(new URL("/", request.url));
    }
    return NextResponse.next();
}

// APRES
if (isPublicRoute(pathname)) {
    return NextResponse.next();
}
```

Impact : un utilisateur deja connecte pourra voir la page /login (UX mineure).
Pour garder la redirection UX, la page login peut elle-meme verifier via fetch
`/api/auth/me` et rediriger cote client si l'utilisateur est authentifie.

### Option B — Nettoyer le cookie dans le middleware
Dans le middleware, au lieu de rediriger, supprimer le cookie et laisser passer :

```typescript
if (isPublicRoute(pathname)) {
    return NextResponse.next();
}

const sessionToken = request.cookies.get(SESSION_COOKIE)?.value;
if (!sessionToken) {
    // ... redirect to /login
}

// Optionnel: on ne peut pas valider en DB depuis Edge,
// mais on peut au moins ne pas rediriger /login → /
```

### Option C — Redirect cote client dans la page login
Garder le middleware tel quel mais ajouter une verification cote client dans LoginPage :

```typescript
// Dans src/app/login/page.tsx
useEffect(() => {
    fetch("/api/auth/me")
        .then((res) => { if (res.ok) router.push("/"); })
        .catch(() => {});
}, []);
```

**Recommandation : Option A** — c'est la plus simple et la plus robuste.
