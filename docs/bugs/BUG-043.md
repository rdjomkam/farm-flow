## Description
On mobile browsers (not PWA standalone), the bottom navigation jitters up/down as the URL bar shows/hides during scroll. Additionally, the safe-area padding region below the nav (home-indicator zone on iOS) renders transparent — the underlying page content bleeds through.

Expected: the nav stays visually pinned to the viewport bottom and the safe-area padding has the same opaque background as the nav.

## Étapes de reproduction
1. Open the app in mobile Safari (iOS) or Chrome Android in a regular browser tab (NOT installed as PWA).
2. Scroll up and down in any page with content taller than the viewport (`/`, `/vagues`, `/releves`).
3. Observe: the bottom nav bounces during the URL bar collapse/expand transition.
4. Observe on iOS with a notched device: the ~34px area below the nav icons is transparent — page content is visible through it.

## Cause racine
See `docs/analysis/pre-analysis-bug-043.md`. Two combined causes:

**A — Jittering:** `position: fixed` on mobile browsers anchors to the **visual viewport**, not the layout viewport. When the URL bar collapses, the viewport changes height and the browser re-paints the fixed element during the transition. Without GPU layer promotion (`translateZ(0)` / `will-change: transform`), this re-paint is visible.

**B — Transparent safe-area:** The nav has `bg-card` and `pb-[env(safe-area-inset-bottom)]` (structurally correct), but there is no global backdrop enforcing `var(--card)` opacity underneath. When the URL bar animates and briefly exposes the gutter, or when the nav re-anchors, a strip of `<html>`/`<body>` background (`var(--surface-0)` `#f7fafa`) peeks through — perceived as transparency.

## Fix
- [x] Add `[transform:translateZ(0)] will-change-transform` to all three bottom-nav variants (farm, ingenieur, skeleton) so iOS/Android use the composited layer and avoid re-layout during URL-bar animation
- [x] Add mobile-only `@media (max-width: 767px)` rule in `src/app/globals.css` painting a `var(--card)` backdrop in the `env(safe-area-inset-bottom)` band so the home-indicator area is always opaque
- [x] Unit test: snapshot nav className list + grep globals.css for the new media query

## Correctif v2 — 2026-04-23 (post-utilisateur : "safe area in the browser is transparent. why?")

### Pourquoi le v1 ne suffisait pas
Le v1 peignait un `linear-gradient` sur `body` avec `background-attachment: fixed`. **iOS Safari ignore `background-attachment: fixed`** (documenté MDN, historique pour raisons de performance) et Chrome Android a un support incohérent. Le dégradé se peignait donc dans la boîte naturelle du body (en haut du document) et jamais dans la bande visuelle du viewport — aucun effet visible sur mobile réel.

### Fix v2 appliqué
Remplacement du `linear-gradient`+`background-attachment:fixed` par un pseudo-élément `position:fixed` dédié — compatible iOS/Android :

```css
@media (max-width: 767px) {
  html::after {
    content: "";
    position: fixed;
    left: 0;
    right: 0;
    bottom: 0;
    height: env(safe-area-inset-bottom);
    background: var(--card);
    pointer-events: none;
    z-index: 39;  /* juste sous la nav (z-40) */
  }
}
```

Fichiers modifiés v2 :
- `/Users/ronald/project/dkfarm/farm-flow/src/app/globals.css` — `linear-gradient` retiré, pseudo-élément `html::after` ajouté
- `/Users/ronald/project/dkfarm/farm-flow/src/components/layout/__tests__/bottom-nav.test.tsx` — assertions renforcées (`html::after`, `position: fixed`, interdiction de `background-attachment: fixed`)

Vérifications v2 :
- `npx vitest run src/components/layout/__tests__/bottom-nav.test.tsx` → **4/4 pass**
- `npx next build --webpack` → **succès**
- Live preview mobile (375×812) : `html::after` résolu avec `position: fixed`, `bottom: 0`, `height: env(safe-area-inset-bottom)`, `background: var(--card)`, `z-index: 39`. Aucune règle `background-attachment: fixed` résiduelle dans la CSS chargée.

### Leçon
`background-attachment: fixed` est activement ignoré par iOS Safari et inconsistant sur Chrome Android. Tout "fixed backdrop" mobile DOIT passer par un pseudo-élément `position: fixed` ou un élément réel en `position: fixed`. Jamais se fier à `background-attachment: fixed` sur mobile.
