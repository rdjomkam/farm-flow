# BUG-046 — Formulaire de relevé en mode lot d'alevins bloqué sur type TRI uniquement

**Sévérité :** Haute
**Détecté par :** @claude (test browser dev server avec lotAlevinsId=lot_02)
**Sprint :** 10
**Fichier(s) :**

- src/components/releves/releve-form-fields.tsx
- src/lib/releve-form-validation.ts
- src/lib/releve-form-dto.ts
- src/app/(farm)/releves/nouveau/page.tsx
- src/components/releves/form-tri.tsx

## Description

Quand l'utilisateur arrive sur la page "Nouveau relevé" depuis un lot d'alevins (`?lotAlevinsId=lot_02`), le formulaire fonctionne correctement pour le type TRI mais échoue pour tous les autres types (BIOMETRIE, MORTALITE, ALIMENTATION, etc.).

Ce qui se passe : changer le type de relevé vers BIOMETRIE ou MORTALITE provoque la perte de `lotAlevinsId` (perdu à la réinitialisation des champs), la section Vague/Bac reste visible et obligatoire, et le submit envoie `vagueId`/`bacId` vides ou redirige incorrectement vers `/vagues/undefined`.

Ce qui devrait se passer : en mode lot d'alevins, TOUS les types de relevé doivent fonctionner. La section Vague doit être masquée pour tous les types. Un sélecteur de bac pré-rempli avec le bac courant du lot doit être affiché. Le submit doit envoyer `lotAlevinsId` (sans `vagueId`) et rediriger vers `/reproduction/lots/<id>`.

## Étapes de reproduction

1. Naviguer vers `/reproduction/lots/lot_02`
2. Cliquer sur "Nouveau relevé" (bouton FAB ou action menu)
3. L'URL contient `?lotAlevinsId=lot_02&typeReleve=TRI`
4. Changer le type vers BIOMETRIE
5. Observer : la section Vague/Bac réapparaît comme obligatoire, les champs spécifiques BIOMETRIE s'affichent mais la validation échoue sur vagueId/bacId manquants

## Cause racine

Cinq verrous côté client :
1. `getEmptyFields(type)` : `lotAlevinsId` injecté uniquement dans le cas TRI via discriminated union — perdu lors du changement de type
2. `isTriWithLot` dans `releve-form-fields.tsx` : ne masque la section Vague/Bac que pour TRI
3. `validateReleveForm` : exige `vagueId`/`bacId` sauf si `TRI + lotAlevinsId`
4. `buildReleveDTO` : bloc spécial `triBase` pour lotAlevinsId uniquement dans TRI
5. Redirection post-submit toujours vers `/vagues/${vagueId}`
