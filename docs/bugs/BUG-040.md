# BUG-040 — Bac manquant dans la liste des relevés après calibrage
**Sévérité :** Haute
**Détecté par :** @user (signalement en production)
**Sprint :** Bugfixing
**Fichier(s) :**
- `src/app/api/bacs/route.ts` (primaire)
- `src/lib/queries/vagues.ts`
- `src/lib/queries/calibrages.ts`

## Description
Après un calibrage redistribuant les poissons vers 3 nouveaux bacs d'une vague,
l'un de ces bacs n'apparaît pas dans la liste des bacs disponibles au moment
de créer un relevé pour cette vague — alors que le calibrage lui-même l'a
accepté comme destination valide.

**Ce qui se passe vs ce qui devrait se passer :**
- Le calibrage accepte N bacs comme destinations.
- Le formulaire de relevé pour la même vague n'affiche qu'un sous-ensemble de
  ces N bacs.
- Attendu : les deux écrans listent **le même ensemble** de bacs pour une vague.

## Étapes de reproduction
1. Créer/ouvrir une vague `EN_COURS` dont certains bacs ont `Bac.vagueId`
   renseigné **sans** `AssignationBac` active correspondante (ou inversement —
   le bug est symétrique).
2. Effectuer un calibrage vers ces bacs.
3. Naviguer vers "Nouveau relevé" → sélectionner la vague.
4. Observer : certains bacs destination du calibrage sont absents de la liste.

## Cause racine

**Incohérence de lecture dual-source** entre les deux écrans sur le modèle
ADR-043 (bac↔vague stocké à la fois sur `Bac.vagueId` et `AssignationBac`) :

### `GET /api/bacs?vagueId=X` (`src/app/api/bacs/route.ts:32-72`)
Lit `AssignationBac` en PRIMAIRE, et ne déclenche le fallback vers
`Bac.vagueId` **que si ZÉRO assignation active n'existe** pour la vague. Si
une assignation existe mais qu'un autre bac de la vague n'est présent que via
`Bac.vagueId`, ce bac est **silencieusement masqué**.

### `getVagueById` (`src/lib/queries/vagues.ts:41-72`) — utilisé par le form calibrage
Lit `Bac.vagueId` en PRIMAIRE (`vague.bacs`), fallback vers les assignations
uniquement si `vague.bacs` est vide. Le symétrique du bug précédent.

### `createCalibrage` (`src/lib/queries/calibrages.ts:197-243`)
Tente d'écrire sur `AssignationBac` via `updateMany`/`findFirst`+`update`.
Si aucune `AssignationBac` active n'existe pour un bac destination, l'écriture
est **silencieusement ignorée** — aucun `create` défensif. Aggrave l'état
incohérent.

## Évidence en base
Requête de diagnostic exécutée le 2026-04-20 sur la base locale :

```sql
SELECT b.id, b.nom, b."vagueId"
FROM "Bac" b
WHERE b."vagueId" IS NOT NULL
  AND NOT EXISTS (
    SELECT 1 FROM "AssignationBac" a
    WHERE a."bacId" = b.id AND a."vagueId" = b."vagueId" AND a."dateFin" IS NULL
  );
```
Résultat : `bac_04` (Etang A) a `Bac.vagueId = vague_02` alors que son
`AssignationBac` est fermée (`dateFin = 2025-12-22`). La vague est
`TERMINEE` — indice que `cloturerVague` ferme l'assignation mais oublie de
nullifier `Bac.vagueId` (bug secondaire).

En production, le même défaut d'invariant peut exister sur des vagues
`EN_COURS`, ce qui reproduit exactement le symptôme signalé.

## Fix
Voir plan approuvé `/Users/ronald/.claude/plans/humble-watching-hammock.md` et
le pipeline BUGFIX (pre-analyst → developer → tester → code-reviewer →
knowledge-keeper) coordonné par `@project-manager`.

Grandes lignes :
1. **Fix principal** — `GET /api/bacs?vagueId` : UNION des deux sources.
2. **Symétrique** — `getVagueById`, `getVagueByIdWithReleves` : idem.
3. **Durcissement** — `createCalibrage` crée une `AssignationBac` manquante
   au lieu de l'ignorer silencieusement.
4. **Migration data** — backfill de `AssignationBac` pour tout
   `Bac.vagueId` non null sans assignation active.
5. **Bug secondaire** — `cloturerVague` doit nullifier `Bac.vagueId` en même
   temps qu'il ferme les assignations.
