# CR-009 — Roles dynamiques par site (remplacement de l'enum Role)

**Type :** Change Request
**Severite :** Haute
**Demande par :** @ronald
**Date :** 2026-03-10
**Sprint cible :** Sprint 9

## Probleme

Le systeme de permissions actuel utilise un enum `Role` fige (ADMIN, GERANT, PISCICULTEUR) sur `SiteMember`. Cela pose plusieurs problemes :

1. **Rigidite** : impossible de creer des roles personnalises (ex: "Responsable stock", "Technicien terrain")
2. **Permissions non ajustables** : les permissions par defaut d'un GERANT sont identiques sur tous les sites — un admin ne peut pas retirer `STOCK_GERER` a un GERANT specifique sans passer par la modification individuelle des permissions
3. **Escalade entre pairs** : un GERANT avec `MEMBRES_GERER` peut actuellement degrader les permissions d'un autre GERANT (bug de securite identifie lors de l'audit)
4. **Couplage backend/roles** : le backend verifie parfois le nom du role (`callerRole !== Role.ADMIN`) au lieu de verifier uniquement les permissions

## Proposition

Remplacer l'enum `Role` sur `SiteMember` par un modele `SiteRole` dynamique, lie au site. Les roles deviennent des entites creees et configurees par l'admin du site. Le backend ne verifie plus jamais le nom d'un role — uniquement les permissions.

### Nouveau modele `SiteRole`

```
SiteRole
├── id          String (cuid)
├── name        String (ex: "Administrateur", "Technicien terrain")
├── description String?
├── permissions Permission[] (memes 29 valeurs existantes)
├── isSystem    Boolean (true pour les 3 roles par defaut — non supprimables)
├── siteId      FK -> Site
├── members     SiteMember[]
└── @@unique([siteId, name])
```

### Modification de `SiteMember`

```
AVANT                          APRES
─────                          ─────
role        Role (enum)   -->  siteRoleId  String (FK -> SiteRole)
permissions Permission[]  -->  (supprime — permissions sur SiteRole)
```

### 3 roles systeme par defaut (crees a la creation d'un site)

| Role | Permissions | Supprimable | Permissions ajustables |
|------|------------|-------------|----------------------|
| Administrateur | Toutes (29) | Non | Oui |
| Gerant | Toutes sauf SITE_GERER, MEMBRES_GERER (27) | Non | Oui |
| Pisciculteur | VAGUES_VOIR, RELEVES_VOIR, RELEVES_CREER, BACS_GERER, DASHBOARD_VOIR, ALERTES_VOIR (6) | Non | Oui |

L'admin du site peut :
- **Creer** des roles personnalises avec n'importe quel sous-ensemble de permissions
- **Modifier** les permissions d'un role existant (y compris les roles systeme)
- **Supprimer** les roles personnalises (les membres sont reassignes au role Pisciculteur)

### Regles d'anti-escalade

- Un utilisateur ne peut creer/modifier un role qu'avec des permissions qu'il possede lui-meme
- Un utilisateur ne peut modifier le role d'un membre que si ses permissions sont un sur-ensemble de celles du membre cible
- Un utilisateur ne peut pas modifier son propre role

### Super-admin global

Le champ `User.role` (enum ADMIN) est conserve. Un User avec `role = ADMIN` contourne toutes les verifications de permissions sur tous les sites (comportement actuel inchange).

### Enforcement backend

`requirePermission()` ne verifie plus jamais le nom du role. Le flux devient :

```
1. Valider session (requireAuth)
2. Si User.role === ADMIN → bypass (toutes permissions)
3. Charger SiteMember + SiteRole (include siteRole)
4. Lire member.siteRole.permissions
5. Verifier que les permissions requises sont presentes
6. Retourner AuthContext { permissions, siteRoleId, siteRoleName, ... }
```

## Migration des donnees existantes

```sql
-- 1. Creer table SiteRole
-- 2. Inserer 3 roles systeme par Site existant
-- 3. Ajouter colonne siteRoleId (nullable) sur SiteMember
-- 4. UPDATE SiteMember SET siteRoleId = (role ADMIN->Administrateur, GERANT->Gerant, PISCICULTEUR->Pisciculteur)
-- 5. ALTER siteRoleId SET NOT NULL + FK + index
-- 6. DROP colonnes role et permissions de SiteMember
```

La colonne `User.role` (enum Role) et l'enum PostgreSQL `Role` ne sont PAS supprimes — toujours utilises pour le super-admin global.

## Regles

- L'enum `Role` reste uniquement pour `User.role` (super-admin)
- Le backend ne verifie JAMAIS le nom d'un role — uniquement les `Permission[]`
- Chaque nouveau site recoit 3 roles systeme automatiquement
- Les roles systeme ne peuvent pas etre supprimes mais leurs permissions sont ajustables
- Anti-escalade : on ne peut accorder que des permissions qu'on possede soi-meme
- Suppression de role : les membres sont reassignes au role systeme Pisciculteur
