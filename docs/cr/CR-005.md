# CR-005 — Auto-set date+heure du Relevé côté backend

**Type :** Change Request
**Sévérité :** Moyenne
**Demandé par :** @project-manager (retour utilisateur)
**Date :** 2026-03-09
**Sprint cible :** À planifier par @project-manager

## Problème

Le champ `Releve.date` est actuellement saisi manuellement par l'utilisateur via un `<input type="date">` (date seule, sans heure). La valeur est envoyée au format "YYYY-MM-DD" à l'API et stockée comme minuit UTC. Cela pose deux problèmes :

1. **Saisie inutile** : la date est presque toujours "aujourd'hui" — le picker ajoute une étape superflue au formulaire mobile
2. **Perte de l'heure** : impossible de savoir à quelle heure un relevé a été effectué (utile pour les relevés d'alimentation multiples dans la journée)

### Impact concret

- Sur mobile 360px, le champ date occupe de l'espace sans valeur ajoutée
- Deux relevés ALIMENTATION le même jour sont indistinguables (même date minuit UTC)
- L'affichage "09/03/2026" ne montre pas l'heure

## Détails des modifications

### 1. `releve-form-client.tsx` — Supprimer le champ date

**Supprimer :**
- L'état : `const [date, setDate] = useState(new Date().toISOString().split("T")[0])`
- Le champ input date (lignes ~221-229)
- La propriété `date` dans le payload `body` envoyé au `fetch`

### 2. `releves/route.ts` — Supprimer la validation date

**Supprimer :**
- Le bloc de validation `if (!date || isNaN(Date.parse(date)))` (lignes ~61-65)
- La déstructuration de `date` depuis `body`
- La propriété `date` de l'objet `base` passé à la query

### 3. `queries/releves.ts` — Timestamp serveur

**Remplacer :**
```typescript
// Avant
date: new Date(data.date),

// Après
date: new Date(),
```

### 4. Affichage — Ajouter l'heure

**Remplacer :**
```typescript
// Avant

// Après
new Date(releve.date).toLocaleString("fr-FR", { dateStyle: "long", timeStyle: "short" })
```

### 5. `api.ts` — Nettoyer le DTO

**Supprimer** la propriété `date: string` de l'interface `CreateReleveBase`.

### 6. Tests — Adapter les payloads

- Supprimer `date: "2026-03-09"` de tous les payloads de test POST/PUT
- Ajouter une assertion : `expect(new Date(body.date).getTime()).toBeCloseTo(Date.now(), -3)`

## Rétrocompatibilité

- **Non rétrocompatible** : le champ `date` n'est plus accepté dans le payload API
- Les relevés existants en base ne sont pas impactés (leur date reste inchangée)
- Les clients envoyant `date` dans le body recevront une erreur ou le champ sera ignoré (au choix de l'implémentation)
- Recommandation : ignorer silencieusement `date` si présent dans le body (tolérance)
