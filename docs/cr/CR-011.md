## Probleme

L'application ne permet pas de comparer objectivement les aliments ou marques utilisees. L'utilisateur :

1. **Ne sait pas quel aliment est le plus rentable** — un aliment cher peut avoir un meilleur FCR et couter moins par kg de poisson produit
2. **Ne peut pas comparer les marques** — pas de vue croisee aliment × performance (FCR, croissance, mortalite)
3. **Achete a l'aveugle** — pas de donnees pour negocier avec les fournisseurs ou justifier un changement de marque
4. **Ne peut pas detecter une degradation de qualite** — si un lot d'aliment est defectueux, c'est invisible

C'est l'argument SaaS le plus fort : aucun concurrent (Manolin, KAMAHU, AquaTracker, FishFarmFeeder) ne propose de comparaison de marques d'aliment avec des metriques financieres pour les petites exploitations francophones.

## Metriques a implementer

### Tier 1 — Essentielles (MVP)

| # | Metrique | Nom FR | Formule | Pourquoi c'est important |
|---|----------|--------|---------|--------------------------|
| 1 | **FCR par aliment** | Indice de conversion par aliment | `totalQuantiteAliment(produitX) / gainBiomasse(vaguesOuProduitXUtilise)` | Comparaison directe entre marques : "Aliment A = FCR 1.5 vs Aliment B = FCR 2.0" |
| 2 | **Cout par kg de gain** | Cout alimentaire par kg de croissance | `(quantite × prixUnitaire) / gainBiomasse` | ROI reel : un aliment cher avec bon FCR peut couter MOINS par kg de poisson. Ex: Aliment A a 2000 CFA/kg avec FCR 1.5 = 3000 CFA/kg gain. Aliment B a 1500 CFA/kg avec FCR 2.2 = 3300 CFA/kg gain. Aliment A est meilleur malgre son prix |
| 3 | **SGR par aliment** | Croissance specifique par aliment | SGR pendant la periode d'utilisation de chaque aliment | Vitesse de croissance : "Avec l'aliment X, les poissons grandissent 20% plus vite" |
| 4 | **Taux de survie associe** | Survie pendant utilisation | Taux de survie pendant les periodes ou l'aliment est utilise | Correlation mortalite/aliment : "Depuis le passage a l'aliment Y, la mortalite a baisse de 3%" |
| 5 | **Quantite totale utilisee** | Consommation totale | `SUM(ReleveConsommation.quantite)` par produit | Vision globale : quel aliment est le plus utilise |
| 6 | **Cout total par aliment** | Depense totale | `SUM(quantite × prixUnitaire)` par produit | Budget : ou va l'argent |

### Tier 2 — Differenciateurs (post-MVP)

| # | Metrique | Formule | Valeur business |
|---|----------|---------|-----------------|
| 7 | **PER (Efficacite proteique)** | `gainPoids(g) / proteineConsommee(g)` | Valide si aliments riches en proteines valent leur prix. Necessite d'ajouter un champ `tauxProteine` sur Produit (optionnel) |
| 8 | **Efficacite alimentaire (FE)** | `(gainBiomasse / alimentConsomme) × 100` | Version % intuitive du FCR, plus facile a comprendre pour les non-techniciens |
| 9 | **Part aliment dans cout total** | `coutAliment / coutTotalVague × 100` | Sensibilite : "L'aliment represente 45% de vos couts — une amelioration de 10% du FCR economise X CFA" |
| 10 | **Impact financier du changement** | `(ancienFCR - nouveauFCR) × prixAliment × productionCible` | Simulateur : "Si vous passez a l'aliment X, vous economisez Y CFA par cycle" |

### Tier 3 — Avancees (optionnel)

| # | Metrique | Formule | Valeur business |
|---|----------|---------|-----------------|
| 11 | **Classement fournisseurs** | Score composite pondere (FCR 40%, cout/kg gain 30%, survie associee 20%, regularite 10%) | Negotiation fournisseurs : "Voici les donnees prouvant que votre aliment sous-performe" |
| 12 | **Tendance FCR par lot** | FCR par periode d'achat du meme produit | Detection degradation qualite : "Le dernier lot de l'aliment X a un FCR 15% pire" |
| 13 | **Correlation aliment × qualite eau** | Regression mortalite vs type aliment + parametres eau | Separer l'effet aliment de l'effet environnement |

## Logique de calcul detaillee

### Comment associer un aliment a une performance ?

Le chainage des donnees est :

```
ReleveConsommation (produitId, quantite)
  → Releve (bacId, vagueId, date)
    → Vague (nombreAlevins, dateDebut)
    → Bac (volume)
    → Autres Releves de la meme vague/bac (BIOMETRIE, MORTALITE, COMPTAGE)
```

**Algorithme pour FCR par aliment :**
1. Pour un `produitId` donne, recuperer tous les `ReleveConsommation` avec ce produit
2. Grouper par `vagueId` (via le `Releve` parent)
3. Pour chaque vague : sommer les quantites consommees de ce produit
4. Recuperer le gain de biomasse de la vague sur la meme periode
5. FCR = quantiteTotale / gainBiomasse

**Algorithme pour cout par kg de gain :**
1. Meme base que FCR
2. Multiplier la quantite par `Produit.prixUnitaire`
3. Diviser par le gain de biomasse

**Cas multi-aliments :** Si une vague utilise 2 aliments en parallele, le FCR par aliment est calcule en repartissant le gain de biomasse proportionnellement a la quantite de chaque aliment (methode standard industrie).

## Proposition d'implementation

### 1. Types TypeScript (`src/types/calculs.ts`)

```typescript
interface AnalytiqueAliment {
  produitId: string
  produitNom: string
  fournisseurNom: string | null
  categorie: string           // ALIMENT
  prixUnitaire: number        // CFA/kg
  // Metriques agregees
  quantiteTotale: number      // kg total utilise
  coutTotal: number           // CFA total depense
  nombreVagues: number        // nb de vagues ou utilise
  fcrMoyen: number | null     // FCR moyen pondere
  sgrMoyen: number | null     // SGR moyen pondere
  coutParKgGain: number | null // CFA par kg de biomasse gagnee
  tauxSurvieAssocie: number | null // survie moyenne pendant utilisation
}

interface ComparaisonAliments {
  siteId: string
  periode: { debut: Date; fin: Date } | null
  aliments: AnalytiqueAliment[]
  meilleurFCR: string | null      // produitId
  meilleurCoutKg: string | null   // produitId
  meilleurSGR: string | null      // produitId
  recommandation: string | null   // texte auto-genere
}

interface DetailAliment extends AnalytiqueAliment {
  parVague: {
    vagueId: string
    vagueNom: string
    quantite: number
    fcr: number | null
    sgr: number | null
    coutParKgGain: number | null
    periode: { debut: Date; fin: Date }
  }[]
  evolutionFCR: { date: Date; fcr: number }[]  // pour graphique tendance
}
```

### 2. Queries d'agregation (`src/lib/queries/analytics.ts`)

Ajouter dans le meme fichier cree par CR-010 :

```
getComparaisonAliments(siteId, filters?): Promise<ComparaisonAliments>
  → Liste tous les produits ALIMENT utilises sur le site
  → Calcule FCR, cout/kg gain, SGR, survie pour chacun
  → Classe par cout/kg gain (le plus rentable d'abord)
  → Genere une recommandation textuelle

getDetailAliment(siteId, produitId): Promise<DetailAliment>
  → Metriques detaillees d'un aliment specifique
  → Ventilation par vague
  → Evolution du FCR dans le temps (pour detecter degradation)

getSimulationChangementAliment(siteId, ancienProduitId, nouveauProduitId, productionCible): Promise<SimulationResult>
  → Compare 2 aliments et calcule l'economie projetee
  → "Si vous passez de A a B pour 1000kg de production, vous economisez X CFA"
```

```
"Sur vos 3 dernieres vagues, l'aliment 'Raanan Fish Feed 42%' (fournisseur: AquaFeed Cameroun)
```

## Cas d'utilisation concrets (argumentaire SaaS)

### 1. Pisciculteur individuel
"Mon fournisseur dit que son aliment est le meilleur. Avec Farm-Flow, je vois que son FCR est 1.9 alors que le concurrent est a 1.5. Je change et j'economise 150 000 CFA par cycle."

### 2. Cooperative de pisciculteurs
"On achete en groupe. Farm-Flow nous montre quel aliment donne les meilleurs resultats sur NOS bassins, avec NOS conditions. On negocie mieux avec les fournisseurs."

### 3. Fournisseur d'aliments
"Je prouve a mes clients que mon aliment premium a un FCR de 1.4. Les donnees Farm-Flow le confirment. Je peux justifier mon prix plus eleve."

### 4. ONG / Projet de developpement
"On finance 50 pisciculteurs. Farm-Flow nous montre quels aliments recommander et ou concentrer la formation."
