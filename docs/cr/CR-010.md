# CR-010 — Analytiques par bac (Tank-Level Analytics)

**Type :** Change Request
**Severite :** Haute
**Demande par :** @ronald
**Date :** 2026-03-10
**Sprint cible :** Sprint 9 ou 10

## Probleme

Actuellement, tous les indicateurs (taux de survie, FCR, SGR, biomasse) sont calcules au niveau de la **vague** uniquement (`getIndicateursVague()` dans `src/lib/queries/indicateurs.ts`). Il est impossible de :

1. **Comparer les bacs** d'une meme vague — un bac peut avoir 60% de survie pendant qu'un autre a 90%, sans que l'utilisateur le sache
2. **Identifier les bacs problematiques** — pas de detection precoce des sous-performances
3. **Optimiser la densite** — pas de correlation densite/performance par bac
4. **Calculer la rentabilite par bac** — impossible de savoir quel bac rapporte le plus

C'est un manque critique pour vendre l'app en SaaS : les concurrents (Manolin, AquaTracker) ne proposent pas d'analytiques par bac pour les petites exploitations francophones.

## Donnees deja disponibles

Toutes les donnees brutes existent. Chaque `Releve` a un `bacId` obligatoire :

| Donnee | Modele / Champ | Pret |
|--------|---------------|------|
| Poids moyen par bac | `Releve` (BIOMETRIE, bacId, poidsMoyen) | Oui |
| Taille moyenne par bac | `Releve` (BIOMETRIE, bacId, tailleMoyenne) | Oui |
| Mortalite par bac + cause | `Releve` (MORTALITE, bacId, nombreMorts, causeMortalite) | Oui |
| Quantite aliment par bac | `Releve` (ALIMENTATION, bacId, quantiteAliment) | Oui |
| Qualite eau par bac | `Releve` (QUALITE_EAU, bacId, temperature, ph, oxygene, ammoniac) | Oui |
| Comptage par bac | `Releve` (COMPTAGE, bacId, nombreCompte) | Oui |
| Consommation produit | `ReleveConsommation` (releveId -> Releve.bacId, produitId, quantite) | Oui |
| Volume du bac | `Bac.volume` | Oui |
| Nombre initial (vague) | `Vague.nombreAlevins` | Oui |

**Il n'y a aucune migration de schema necessaire.** Seules les fonctions de calcul, queries d'agregation, routes API et pages UI sont a creer.

## Metriques a implementer

### Tier 1 — Essentielles (MVP)

| # | Metrique | Nom FR | Formule | Benchmark Clarias |
|---|----------|--------|---------|-------------------|
| 1 | Taux de survie | Taux de survie | `(dernierComptage / nombreInitialBac) × 100` | Cible >90%, bon 85-90%, acceptable 80-85% |
| 2 | FCR | Indice de conversion alimentaire | `totalAlimentDistribue(kg) / gainBiomasse(kg)` | Cible <1.5, bon 1.5-1.8, acceptable 1.8-2.2 |
| 3 | SGR | Taux de croissance specifique | `[ln(poidsFinal) - ln(poidsInitial)] / jours × 100` (% par jour) | Cible >2%/j, bon 1.5-2%, acceptable 1-1.5% |
| 4 | Biomasse | Biomasse totale | `(poidsMoyen × nombreVivants) / 1000` (kg) | Depends de la taille du bac |
| 5 | Densite | Densite de biomasse | `biomasse(kg) / volume(m³)` | Cible ~7 poissons/m³ |
| 6 | Mortalite | Taux de mortalite | `totalMorts / nombreInitialBac × 100` | Cible <5%, acceptable 5-10% |
| 7 | Gain quotidien | Gain de biomasse quotidien | `(biomasseFin - biomasseDebut) / jours` (kg/j) | Variable selon densite |

### Tier 2 — Differenciateurs (post-MVP)

| # | Metrique | Formule | Valeur business |
|---|----------|---------|-----------------|
| 8 | Cout aliment / kg produit | `coutTotalAliment / gainBiomasse` | Rentabilite par bac |
| 9 | Score qualite eau | Composite normalise (temp, pH, O2, NH3) par rapport aux seuils optimaux | Correlation avec performance |
| 10 | Recolte estimee | `poidsMoyen × nombreVivants` au prix du marche | Prevision de vente |
| 11 | Duree du cycle | `dateRecolte - dateEmpoissonnement` | Rotation du capital |
| 12 | ROI du bac | `(revenuRecolte - coutsTotaux) / coutsTotaux × 100` | Metrique ultime |

### Tier 3 — Avancees (optionnel)

| # | Metrique | Formule | Valeur business |
|---|----------|---------|-----------------|
| 13 | TGC | `(poidsFin^(1/3) - poidsInit^(1/3)) / (degres-jours × 10)` | Comparaison inter-saisons (normalise par temperature) |
| 14 | Frequence traitements | Nb evenements maladie par bac par cycle | Detection bacs chroniquement problematiques |

## Proposition d'implementation

### 1. Fonctions de calcul pures (`src/lib/calculs.ts`)

Ajouter les fonctions suivantes. Elles reprennent les signatures existantes (`calculerTauxSurvie`, `calculerFCR`, `calculerSGR`, `calculerBiomasse`) mais ajoutent les nouvelles :

```
calculerDensite(biomasse: number, volume: number): number | null
```

Les fonctions existantes (`calculerTauxSurvie`, `calculerFCR`, `calculerSGR`, `calculerBiomasse`) restent inchangees — elles sont deja reutilisables pour un bac individuel.

### 2. Types TypeScript (`src/types/calculs.ts`)

```typescript
interface IndicateursBac {
  bacId: string
  bacNom: string
  vagueId: string
  volume: number              // m³
  // Tier 1
  tauxSurvie: number | null   // %
  fcr: number | null
  sgr: number | null           // %/jour
  biomasse: number | null      // kg
  densite: number | null       // kg/m³
  tauxMortalite: number | null // %
  gainQuotidien: number | null // kg/jour
  // Metadata
  dernierReleve: Date | null
  nombreReleves: number
}

interface ComparaisonBacs {
  vagueId: string
  vagueNom: string
  bacs: IndicateursBac[]
  meilleurFCR: string | null     // bacId
  meilleurSurvie: string | null  // bacId
  alertes: AlerteBac[]
}

interface AlerteBac {
  bacId: string
  type: 'SURVIE_BASSE' | 'FCR_ELEVE' | 'MORTALITE_HAUTE' | 'DENSITE_ELEVEE'
  message: string
  valeur: number
  seuil: number
}
```

### 3. Queries d'agregation (`src/lib/queries/analytics.ts` — NOUVEAU)

```
getIndicateursBac(siteId, vagueId, bacId): Promise<IndicateursBac>
  → Agrege tous les releves d'un bac pour calculer ses indicateurs

getComparaisonBacs(siteId, vagueId): Promise<ComparaisonBacs>
  → Calcule les indicateurs de TOUS les bacs d'une vague et les compare
  → Identifie le meilleur/pire sur chaque metrique
  → Genere les alertes

getHistoriqueBac(siteId, bacId): Promise<HistoriqueBac>
  → Performance du bac a travers les differentes vagues (cycles successifs)
```

Patron a suivre : `src/lib/queries/indicateurs.ts` (existant) pour la structure des requetes Prisma avec agregation.
