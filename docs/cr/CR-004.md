# CR-004 — Lier les Relevés au Stock (Consommation Produits + Déduction Auto)

**Type :** Change Request
**Sévérité :** Haute
**Demandé par :** @project-manager (analyse fonctionnelle)
**Date :** 2026-03-09
**Sprint cible :** À planifier par @project-manager

## Problème

Les relevés de types ALIMENTATION, QUALITE_EAU et MORTALITE consomment des produits (aliments, intrants chimiques, médicaments) mais n'ont **aucune connexion avec le système de stock**. Les utilisateurs ne peuvent pas sélectionner les produits utilisés lors d'un relevé, et aucun MouvementStock n'est créé pour déduire l'inventaire.

### Impact concret

- **Double saisie** : l'utilisateur saisit `quantiteAliment` dans le relevé PUIS doit manuellement créer un mouvement de stock SORTIE
- **Incohérence stock** : le stock affiché ne reflète pas la consommation réelle si l'utilisateur oublie la saisie manuelle
- **Pas de traçabilité** : impossible de savoir quel relevé a consommé quel produit
- **Données partielles** : QUALITE_EAU et MORTALITE ne capturent pas du tout les produits de traitement utilisés

## Analyse de l'écart (Gap Analysis)

| Élément | État actuel | État souhaité |
|---------|-------------|---------------|
| Releve → Produit | Aucun lien | Lien N:N via table bridge |
| ALIMENTATION | `quantiteAliment` (Float) sans lien produit | Sélection produit(s) ALIMENT + quantité → déduction stock |
| QUALITE_EAU | Mesures pH/temp/O2 uniquement | + produits INTRANT optionnels utilisés pour traitement |
| MORTALITE | `nombreMorts` + `causeMortalite` | + produits INTRANT optionnels (antibiotique, désinfectant) |
| MouvementStock | Créé manuellement | Créé automatiquement à la validation du relevé |
| Traçabilité | MouvementStock sans lien relevé | `releveId` sur MouvementStock pour remonter à la source |

## Types de relevé et interactions stock

| Type relevé | Catégorie produit | Exemple concret | Obligatoire ? |
|-------------|-------------------|-----------------|---------------|
| ALIMENTATION | `ALIMENT` | "Aliment Commercial 35%" — 5.5 kg | Optionnel (rétrocompatible) |
| QUALITE_EAU | `INTRANT` | "Correcteur pH" — 0.5 L | Optionnel |
| MORTALITE | `INTRANT` | "Antibiotique Oxytetracycline" — 100g | Optionnel |

## Proposition technique

### 1. Nouveau modèle — `ReleveConsommation` (table bridge)

```prisma
model ReleveConsommation {
  id        String   @id @default(cuid())
  releveId  String
  releve    Releve   @relation(fields: [releveId], references: [id], onDelete: Cascade)
  produitId String
  produit   Produit  @relation(fields: [produitId], references: [id])
  quantite  Float
  siteId    String
  site      Site     @relation(fields: [siteId], references: [id])

  @@index([releveId])
}
```

- **Cardinalité** : 1 Releve → N ReleveConsommation (plusieurs produits par relevé)
- **Cascade** : suppression du relevé → suppression des consommations
- **R8** : `siteId` inclus conformément aux règles Phase 2

### 2. Modification `MouvementStock` — ajout `releveId`

```prisma
model MouvementStock {
  // ... champs existants
  releveId  String?
  releve    Releve?  @relation(fields: [releveId], references: [id])
}
```

Permet de remonter de n'importe quel mouvement vers le relevé qui l'a généré.

### 3. Logique transactionnelle

Pour chaque relevé avec consommations, une transaction Prisma atomique :

```
prisma.$transaction([
  1. Créer le Releve
  2. Pour chaque consommation :
     a. Créer ReleveConsommation
     b. Créer MouvementStock (type=SORTIE, releveId=releve.id)
     c. Décrémenter Produit.stockActuel
])
```

**Validation pré-transaction :**
- Vérifier que chaque `produitId` existe et appartient au même `siteId`
- Vérifier que `stockActuel >= quantite` pour chaque produit (empêcher stock négatif)
- Vérifier la catégorie produit vs type de relevé (ALIMENT pour ALIMENTATION, INTRANT pour QUALITE_EAU/MORTALITE)

## Rétrocompatibilité

- **100% rétrocompatible** : le champ `consommations` est optionnel dans le payload API
- Les relevés existants ne sont pas impactés (pas de migration de données)
- Le champ `quantiteAliment` existant sur Releve est conservé (donnée déclarative vs consommation stock)
- Les formulaires fonctionnent sans sélection de produit (comme avant)

## Risques

| Risque | Probabilité | Mitigation |
|--------|-------------|------------|
| Stock négatif si validations concurrentes | Faible | Transaction Prisma avec vérification stock dans la même transaction |
| Complexité formulaire mobile | Moyenne | Product picker en modal/sheet, pas inline |
| Performance N+1 queries | Faible | Include `consommations` avec `produit` dans les queries |
| Migration sur données existantes | Nulle | Nouveau modèle, pas de modification de données existantes |
