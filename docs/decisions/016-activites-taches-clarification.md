# ADR-016 — Clarification de la dualité "Activités planifiées" vs "Mes tâches"

**Statut :** Proposé
**Date :** 2026-03-18
**Auteur :** @architect

---

## Contexte

Un utilisateur d'une ferme supervisée se retrouve face à deux entrées de navigation distinctes qui semblent
redondantes :

- **"Mes tâches"** (`/mes-taches`) — visible dans la bottom nav du rôle PISCICULTEUR
- **"Calendrier" / Planning** (`/planning`) — visible dans le module "Analyse & Pilotage"

Les deux affichent des `Activite` du même modèle Prisma, toutes deux permettent de marquer comme complété
via `CompleterActiviteDialog`. La confusion est réelle et documentée par les retours utilisateurs.

---

## Analyse du code actuel

### Le modèle unique `Activite`

Il n'existe qu'un seul modèle en base de données :

```
model Activite {
  assigneAId   String?        // null = activité du site, non nulle = assignée à quelqu'un
  isAutoGenerated Boolean     // true = moteur de règles, false = créée manuellement
  statut       StatutActivite // PLANIFIEE | TERMINEE | ANNULEE | EN_RETARD
  ...
}
```

Les deux vues ne sont que des **projections filtrées** du même jeu de données :

| Vue | Filtre appliqué | Source de données |
|-----|----------------|-------------------|
| `/mes-taches` | `assigneAId = userId` (tous statuts) | `getAllMyTasks()` |
| `/planning` | date dans fenêtre ±2 mois (tous assignés) | `getActivites()` avec plage de dates |

**Conclusion technique** : "Mes tâches" est strictement un filtre `WHERE assigneAId = currentUser` sur le
même modèle `Activite`. Ce n'est pas un concept séparé.

### Ce qui déclenche la confusion

**Confusion 1 — Même action, deux endroits**

`CompleterActiviteDialog` est rendu dans :
- `ActiviteCard` (utilisé par `ActiviteListClient` dans `/mes-taches`)
- `PlanningClient` (dans le dialog de détail en `/planning`)
- `InstructionViewer` (dans `/mes-taches/[id]`)

Un pisciculteur voit donc le bouton "Compléter" dans `/mes-taches` ET dans `/planning` pour les activités
qui lui sont assignées et qui tombent dans la fenêtre du calendrier.

**Confusion 2 — Sémantique floue**

"Activités planifiées" suggère quelque chose de géré par l'admin (le calendrier du site). "Mes tâches"
suggère une to-do list personnelle. Mais les deux affichent des activités `isAutoGenerated=true` générées
par le moteur de règles, qui ont souvent `assigneAId = null` (non assignées à personne en particulier).

La page `/mes-taches` ne charge que les activités `assigneAId = userId`. Si le moteur de règles génère des
activités sans `assigneAId`, elles ne sont **pas visibles** dans "Mes tâches" mais apparaissent dans le
planning. C'est une incohérence sémantique majeure : des activités auto-générées urgentes peuvent être
invisibles pour le pisciculteur dans son interface principale.

**Confusion 3 — Permission asymétrique**

- La permission `PLANNING_GERER` est requise pour pouvoir compléter une activité (dans `ActiviteCard` et
  `InstructionViewer`).
- Le rôle PISCICULTEUR n'a pas `PLANNING_GERER` par défaut (voir `SYSTEM_ROLE_DEFINITIONS`).
- Résultat : un pisciculteur voit ses tâches dans `/mes-taches` mais **ne peut pas les marquer comme
  complétées** — le bouton "Compléter" n'apparaît jamais.
- Cette permission est pourtant la même que celle requise pour créer/modifier une activité via le calendrier.

**Confusion 4 — Activations packs**

La page `/activations` (module "Packs & Provisioning") affiche les `PackActivation`, un modèle différent.
Elle n'a rien à voir avec les activités d'élevage. Le label "Activations" dans la nav peut être confondu
avec "activités" dans le langage courant.

---

## Options considérées

### Option A — Statu quo (ne rien changer)

Maintenir deux vues séparées avec la confusion actuelle.

**Inconvénients** :
- La confusion utilisateur persiste
- Un pisciculteur ne peut pas compléter ses propres tâches (bug de permission)
- Duplications de logique dans les composants

**Verdict** : Rejeté.

### Option B — Supprimer "Mes tâches", tout mettre dans Planning

Fusionner les deux vues : `/planning` devient le seul endroit pour voir et compléter les activités.

**Inconvénients** :
- Le calendrier est une vue transversale (tout le site), pas personnelle
- Un pisciculteur avec peu de permissions serait noyé dans les activités des autres
- La navigation PISCICULTEUR perdrait son ancrage ("Accueil, Analyse, [??], Échanges")
- Le calendrier nécessite `PLANNING_VOIR`, permission que le pisciculteur n'a pas

**Verdict** : Rejeté. Les deux niveaux de lecture (personnel vs site) ont une valeur réelle.

### Option C — Renommer et clarifier les rôles, corriger les permissions (recommandée)

Traiter les deux vues comme **complémentaires et explicitement différentes** :

- `/mes-taches` = **vue personnelle d'action** : "Ce que je dois faire aujourd'hui/cette semaine"
- `/planning` = **vue de gestion collective** : "Tout ce qui est planifié sur le site ce mois"

Les changements portent sur la sémantique, les permissions et les activités non-assignées.

---

## Recommandation — Option C

### 1. Corriger la permission de complétion pour les pisciculteurs

Le problème fondamental est que `canComplete` est conditionné à `PLANNING_GERER`, une permission de
**gestion** du planning. Compléter une tâche qui m'est assignée est une opération **d'exécution**, pas de
gestion.

**Changement proposé dans `ActiviteCard` et `InstructionViewer`** :

```typescript
// Avant
const canComplete =
  permissions.includes(Permission.PLANNING_GERER) &&
  (activite.statut === StatutActivite.PLANIFIEE || activite.statut === StatutActivite.EN_RETARD);

// Après
const canComplete =
  (permissions.includes(Permission.PLANNING_GERER) ||
   (permissions.includes(Permission.DASHBOARD_VOIR) && activite.assigneAId === currentUserId)) &&
  (activite.statut === StatutActivite.PLANIFIEE || activite.statut === StatutActivite.EN_RETARD);
```

Ou plus proprement : ajouter une permission dédiée `ACTIVITES_COMPLETER` au rôle PISCICULTEUR, distincte
de `PLANNING_GERER` qui reste réservé aux admins/gérants.

### 2. Étendre "Mes tâches" aux activités non assignées du site

Un pisciculteur doit pouvoir voir les activités auto-générées qui concernent son site même si elles n'ont
pas d'`assigneAId`. La requête `getAllMyTasks` doit inclure les activités `assigneAId = null` avec une
section distincte "À faire (non assignées)".

```typescript
// Logique de requête proposée
async function getAllMyTasks(siteId: string, userId: string) {
  // Section 1 : Mes tâches (assignées à moi)
  const mesTaches = await prisma.activite.findMany({
    where: { siteId, assigneAId: userId }
  });

  // Section 2 : Tâches du site non assignées (visibles à tous, PLANIFIEE/EN_RETARD seulement)
  const tachesSite = await prisma.activite.findMany({
    where: {
      siteId,
      assigneAId: null,
      statut: { in: [StatutActivite.PLANIFIEE, StatutActivite.EN_RETARD] }
    }
  });

  return { mesTaches, tachesSite };
}
```

### 3. Différencier clairement les labels et l'intention des deux vues

| Aspect | /mes-taches | /planning |
|--------|-------------|-----------|
| Titre affiché | "Mes tâches" | "Planning" ou "Calendrier" |
| Sous-titre | "Tâches assignées + à faire aujourd'hui" | "Vue du planning du site" |
| Qui la voit | PISCICULTEUR (bottom nav prioritaire) | ADMIN, GERANT (module Analyse & Pilotage) |
| Filtre implicite | assigneAId = moi + non assignées PLANIFIEE/EN_RETARD | tout le site, toutes dates |
| Complétion | Oui (permission exécution) | Oui (permission gestion) |
| Création | Non | Oui (PLANNING_GERER) |
| Vue | Liste priorisée (urgentes d'abord) | Calendrier mensuel |

### 4. Supprimer la duplication de la logique de completion

`CompleterActiviteDialog` doit rester le seul composant pour compléter une activité. Il est déjà utilisé
correctement dans les deux contextes. Pas de changement architectural nécessaire ici.

### 5. Clarifier l'entrée "Activations" dans la nav

Renommer `/activations` en "Historique packs" ou "Packs activés" pour dissiper la confusion avec
"activités". L'item actuel dans `module-nav-items.ts` :

```typescript
// Avant
{ href: "/activations", label: "Activations", icon: ClipboardCheck }

// Après
{ href: "/activations", label: "Historique", icon: History }
// ou
{ href: "/activations", label: "Packs actives", icon: CheckSquare }
```

## Impact sur les fichiers

### Permissions — changements prioritaires

| Fichier | Changement |

| `src/lib/permissions-constants.ts` | Ajouter `ACTIVITES_COMPLETER` au groupe `planning`, l'ajouter pour PISCICULTEUR |
| `src/types/models.ts` | Ajouter `Permission.ACTIVITES_COMPLETER` à l'enum |
| `src/components/activites/activite-card.tsx` | Utiliser la nouvelle permission pour `canComplete` |
| `src/components/activites/instruction-viewer.tsx` | Idem |

### Requête "Mes tâches" — changements importants

| Fichier | Changement |
|---------|-----------|
| `src/lib/queries/activites.ts` | Modifier `getAllMyTasks()` pour retourner `{ mesTaches, tachesSite }` |
| `src/app/mes-taches/page.tsx` | Passer `userId` en plus pour la permission de complétion contextuelle |
| `src/components/activites/activite-list-client.tsx` | Accepter deux groupes de données, afficher section "Du site" |

### Navigation — changements cosmétiques

| Fichier | Changement |
|---------|-----------|
| `src/lib/module-nav-items.ts` | Renommer "Activations" → "Packs actives" dans le module Packs |

### Pas de changements nécessaires

- Le schéma Prisma (`schema.prisma`) : pas de nouveau modèle, `Activite` est déjà correct
- Les API routes (`/api/activites/`) : le endpoint `/complete` reste inchangé
- Le composant `PlanningClient` : il reste la vue collective, pas de changement sémantique

## Flux utilisateur après correction

### Pisciculteur (ferme supervisée)

1. Ouvre l'app sur mobile → bottom nav : Accueil | Analyse | **Mes tâches** | Échanges
2. Tape "Mes tâches" → voit ses activités assignées + les activités non assignées du site
3. Section "En retard" en rouge en haut (urgences)
4. Section "Mes activités" (assigneAId = moi) avec badge "Assignée à moi"
5. Section "À faire sur le site" (activités non assignées, PLANIFIEE/EN_RETARD)
6. Tape une carte → page détail → peut marquer comme complété (permission ACTIVITES_COMPLETER)
7. Ne voit jamais le Planning (pas la permission, pas dans sa nav)

### Admin / Gérant

1. Ouvre l'app → accès au module "Analyse & Pilotage" dans le menu
2. Tape "Calendrier" → vue calendrier mensuelle de tout le site
3. Peut créer, modifier, supprimer, compléter n'importe quelle activité (PLANNING_GERER)
4. Peut aussi aller dans "Mes tâches" si des activités lui sont assignées directement

---

## Décision

Adopter l'Option C avec priorité d'implémentation :

1. **P0** — Corriger la permission `canComplete` pour les pisciculteurs (bug bloquant UX)
2. **P1** — Étendre `getAllMyTasks` aux activités non assignées du site
3. **P2** — Mettre à jour `ActiviteListClient` pour afficher les deux groupes avec distinction visuelle
4. **P3** — Renommer "Activations" → "Packs actives" dans la nav

Les changements P0 et P1 doivent être traités dans le sprint courant car ils constituent des bugs
fonctionnels (un pisciculteur ne peut pas accomplir sa mission primaire).
