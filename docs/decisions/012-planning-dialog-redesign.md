# 012 ‚Äî Planning Activity Detail Dialog Redesign

**Date:** 2026-03-18
**Status:** Proposed
**Author:** @architect

---

## Context

The current activity detail dialog in `src/components/planning/planning-client.tsx` (lines ~394‚Äì528) is functional but visually flat. It uses bare `div` + `text-sm` rows without structure or visual hierarchy. It does not leverage the richer design language developed in Sprint 22‚Äì23 (InstructionViewer, ActiviteCard), and it misses several fields available on `ActiviteWithRelations`: `instructionsDetaillees`, `conseilIA`, `produitRecommande`, `quantiteRecommandee`, `priorite`, `isAutoGenerated`, `phaseElevage`, `regle`.

This ADR defines the target design without touching code.

### Section card pattern (from InstructionViewer, ActiviteCard)
```

```
This is the canonical "card section" used throughout the app. It is the single most reused block in detail views.

### Colored alert block pattern
```
rounded-xl border border-{color}/30 bg-{color}/10 p-4
```
Used for: `conseilIA` (warning), `noteCompletion` when TERMINEE (success), EN_RETARD banner (danger).

### Pill tag pattern (from InstructionViewer meta bar)
```
rounded-full bg-muted px-2 py-0.5 text-xs text-muted-foreground
```
Used for: type label, vague code, bac name, phase d'elevage.

### Primary-tinted pill
```
rounded-full bg-primary/10 px-2 py-0.5 text-xs font-medium text-primary
```
Used for: phase d'elevage when it has semantic meaning.

### Auto-generated pill
```
inline-flex items-center gap-1 rounded-full bg-accent-purple/10 px-2 py-0.5 text-xs text-accent-purple
```
Icon: `<Bot className="h-3 w-3" />`

### Key-value row pattern
```
flex justify-between text-sm
  span.text-muted-foreground   ‚Üê label
  span.font-medium             ‚Üê value
```

### Icon + title section header pattern
```

```

## What Works Well in Existing Components

The `InstructionViewer` (`src/components/activites/instruction-viewer.tsx`) already solves the display problem for the full-page `/mes-taches/[id]` view. The dialog redesign should mirror its section structure, adapted for the dialog's constrained width (100% on mobile, `max-w-lg` on desktop).

Key patterns to import directly:
1. Header card: `rounded-xl border bg-card p-4` with title + description + meta pills bar
2. Colored alert for `conseilIA`: `border-warning/30 bg-warning/10`
3. Colored completion block for `noteCompletion`: `border-success/30 bg-success/10`
4. Instructions section with `<FileText>` icon header
5. Priority config from `activite-card.tsx` (`getPrioriteConfig` function)
6. Type icons from `activite-card.tsx` (`typeActiviteIcons` map)

## DialogHeader Usage

The current dialog uses bare `DialogTitle` without `DialogHeader`. The redesign should keep this structure ‚Äî `DialogHeader` wraps `DialogTitle` with `pb-4` padding, which would add unnecessary whitespace since Section 1 (the header card) already provides the title visually with full context.

**Recommended:** Keep `<DialogTitle>` for accessibility (it is read by screen readers), but visually hide it or make it a small label above the first card section:

```
<DialogTitle className="sr-only">{selectedActivite.titre}</DialogTitle>
```

This satisfies ARIA without duplicating the visible title already shown in Section 1.

---

## Mobile-First Sizing

The `DialogContent` component already enforces:
- Mobile: `inset-0 rounded-none` ‚Äî full screen, scrollable
- Desktop: `md:max-w-lg md:rounded-xl md:max-h-[85vh]`

All cards use `p-4` padding (not `p-6`) to preserve content width on 360px.

The actions footer uses `flex flex-wrap gap-2` so buttons stack gracefully. The CompleterActiviteDialog trigger should be `w-full sm:w-auto` when it is the primary action.

The min-touch-target size of 44px must be preserved on all interactive elements inside the dialog, including the stock link button (already enforced by `min-h-[44px]`).

---

## Color Coding by Activity Type

The `typeActiviteColors` map already in `planning-client.tsx` uses `bg-{color}` for the dot in the legend. The redesigned header card icon uses `iconBgClass` / `iconTextClass` from `getPrioriteConfig` ‚Äî these are independent and must not be confused.

There is no requirement to introduce per-type coloring in the dialog beyond what `getPrioriteConfig` provides via priority. The type is already distinguished by icon.
