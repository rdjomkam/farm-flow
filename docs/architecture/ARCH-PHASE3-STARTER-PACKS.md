## 1. Overview & Context

### 1.1 Phase 3 Scope

Phase 3 introduces **Starter Packs** — a B2C product where DKFarm sells a kit (alevins + feed + health products) bundled with guided farming through the FarmFlow application. The system automatically creates daily/weekly activities for the client, calculates feeding quantities, detects growth phase transitions, and enables remote engineer monitoring with bidirectional observations.

> **Note:** AI advisor integration (Claude API) has been deferred to Phase 4. The rule-based Activity Engine provides comprehensive guided farming coverage without AI dependency.

The five sub-phases are:
- **Sprint 13:** ConfigElevage model + refactoring of hardcoded benchmarks/alerts
- **Sprint 14:** Pack + PackProduit + PackActivation + provisioning engine
- **Sprint 15:** Activity engine (RegleActivite + rule evaluation + CRON)
- **Sprint 16:** Enriched task UX with detailed instructions + projections
- **Sprint 17:** Engineer monitoring dashboard + NoteIngenieur + client observations

### 1.2 Relationship to Existing Phases

**Phase 1 (Sprints 1-5, COMPLETE):** Core CRUD — Bac, Vague, Releve, dashboard, basic calculations in `benchmarks.ts`, `alertes.ts`, `calculs.ts`. These files contain hardcoded thresholds that Phase 3 replaces with configurable values from ConfigElevage.

**Phase 2 (Sprints 6-12, IN PROGRESS):** Authentication, multi-tenancy (Site/SiteMember), stock management (Produit/MouvementStock/Commande), sales/invoicing (Client/Vente/Facture), alevins production, alerts/planning (Activite/Notification/ConfigAlerte), and export. Phase 3 builds directly on the multi-tenancy model (a client gets their own Site) and the Activite model (auto-generated activities).

### 1.3 Key Architectural Decisions Summary

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | ConfigElevage values are COPIED to client site on activation, not referenced | Prevents DKFarm config changes from silently altering active client farms |
| D2 | Activite.userId is NULLABLE for auto-generated activities with `isAutoGenerated=true` | No "system user" concept; auto-generated activities have no human creator |
| D3 | Remove `@unique` on PackActivation.clientSiteId and vagueId | A site can have multiple activations; a vague can be replaced after failure |
| D4 | Activity engine uses event-driven + CRON hybrid | Threshold rules fire on releve creation; recurrent/calendar rules fire on daily CRON |
| D5 | Idempotency via RuleExecution tracking table | Prevents duplicate activity generation on CRON re-runs |
| D6 | New Role: INGENIEUR with read-only cross-site access | Engineers monitor clients without ADMIN write privileges |
| D7 | Template engine uses simple `{placeholder}` string replacement | No Handlebars dependency; limited placeholder vocabulary is sufficient |
| D8 | All day calculations use WAT (UTC+1) timezone | Cameroon operates on WAT; prevents off-by-one day boundary issues |
| D9 | Zod validation for ConfigElevage JSON fields at API layer | PostgreSQL has no native JSON schema validation; enforce at application level |

## 3. Activity Engine Architecture

### 3.1 Hybrid Approach: Event-Driven + CRON

The activity engine operates in two modes:

**Event-driven (immediate):** After a Releve is created via the API, the system evaluates all threshold-based rules for the associated vague:
- `SEUIL_POIDS` — triggered by biometry releves
- `SEUIL_QUALITE` — triggered by water quality releves
- `SEUIL_MORTALITE` — triggered by mortality releves
- `FCR_ELEVE` — triggered after any releve that affects FCR calculation
- `STOCK_BAS` — triggered after stock movement (MouvementStock creation)

**CRON-driven (daily):** A scheduled job evaluates time-based rules for all active vagues:
- `RECURRENT` — based on `intervalleJours` since last execution
- `CALENDRIER` — based on `jourDepuisDebut` matching the current day count
- `JALON` — milestone-based, same as CALENDRIER but with phase awareness

### 3.2 CRON Mechanism

**Selected: Vercel Cron Jobs** (API route triggered by Vercel's scheduler)

| Option | Pros | Cons | Decision |
|--------|------|------|----------|
| Vercel Cron Jobs | Native to deployment platform, no external service, free tier includes daily jobs | Max 1 invocation per path per hour on free plan; 60s timeout on Hobby | **Selected** |
| pg_cron | Runs inside PostgreSQL, no HTTP overhead | Prisma Postgres (prod) may not support pg_cron extensions; hard to debug | Rejected |
| External (cron-job.org, Upstash) | Flexible scheduling, no platform lock-in | External dependency, secret management, additional cost | Rejected (fallback) |

Implementation: an API route at `/api/cron/activites` protected by a bearer token (`CRON_SECRET` env var). Vercel cron config in `vercel.json`:

```json
{
  "crons": [
    {
      "path": "/api/cron/activites",
      "schedule": "0 5 * * *"
    }
  ]
}
```

The CRON runs at 05:00 UTC (06:00 WAT), generating the day's activities before the typical farming workday begins.

**Timeout handling:** The CRON route processes vagues in batches of 20. If approaching the 60s timeout, it returns a 206 Partial response. Vercel does not retry by default, so a secondary invocation at 05:30 UTC handles any remaining vagues using a `lastProcessedAt` cursor stored in a `CronState` key-value record.

### 3.3 Rule Evaluation Flow

```

    v
┌─────────────────────────────────┐

│    WHERE statut = EN_COURS      │
│    AND site.isActive = true     │

               │

```

### 3.4 Idempotency

The `RuleExecution` table is the single source of truth for "has this rule already fired for this vague today?"

**Hash computation by trigger type:**

| Trigger | triggerHash formula |
|---------|-------------------|
| RECURRENT | `sha256(regleId)` — one execution per day per rule |
| CALENDRIER | `sha256(regleId + jourDepuisDebut)` — fires once per specific day |
| JALON | `sha256(regleId + jourDepuisDebut)` |
| SEUIL_POIDS | `sha256(regleId + seuilPoidsMin + seuilPoidsMax)` — fires once per threshold band |
| SEUIL_QUALITE | `sha256(regleId + parametreQualite + comparateur + seuilValeur)` |
| SEUIL_MORTALITE | `sha256(regleId + seuilValeur)` |
| STOCK_BAS | `sha256(regleId + produitCategorie)` — re-fires if stock goes below threshold again after being above |
| FCR_ELEVE | `sha256(regleId + floor(fcrValue * 10))` — fires once per 0.1 FCR increment |

**Re-enabling threshold rules (EC-3.2):** For SEUIL_POIDS and similar rules, the hash encodes the threshold band. Once the metric leaves the band and re-enters it, a new execution is allowed because the previous execution's date differs. To handle the case where a threshold is crossed, un-crossed, and re-crossed on the same day, we accept at most one activity per day per rule per vague.

### 3.5 Template Placeholder Engine

Simple `{placeholder}` string replacement. No nested expressions, no conditionals.

**Available placeholders:**

| Placeholder | Source | Example |
|-------------|--------|---------|
| `{semaine}` | `Math.ceil(jourDepuisDebut / 7)` | `4` |
| `{jour}` | `jourDepuisDebut` | `28` |
| `{poids_moyen}` | Latest biometry `poidsMoyen` (g) | `42.5` |
| `{nombre_vivants}` | `nombreInitial - totalMorts + transferts` | `275` |
| `{taux_survie}` | Calculated % | `91.6` |
| `{quantite_calculee}` | Computed from feeding formula | `540` |
| `{taille}` | From `alimentTailleConfig` lookup | `2mm` |
| `{stock}` | `stockActuel` for the relevant product category | `8500g` |
| `{seuil}` | `seuilAlerte` of the product | `2000g` |
| `{jours_restants}` | `stockActuel / consommationQuotidienne` | `15` |
| `{quantite_recommandee}` | `consommationQuotidienne * stockJoursAlerte * 2` | `5400g` |
| `{produit}` | Product name | `Granule 2mm` |
| `{valeur}` | The measured value that triggered the rule | `6.2` |
| `{taux}` | Mortality or FCR rate that triggered | `1.5` |
| `{fcr}` | Current FCR value | `1.8` |
| `{phase}` | Current PhaseElevage label | `CROISSANCE_DEBUT` |

**Error handling for unresolvable placeholders (EC-3.6):** If a placeholder cannot be resolved (e.g., no biometry data yet), it is replaced with `[non disponible]` and the activity is still created. The `Activite.notes` field records which placeholders could not be resolved.

### 3.6 Deduplication Strategy

Beyond RuleExecution-based idempotency, additional deduplication occurs at the Activite level:

1. Before creating a new auto-generated Activite, check if an identical activity already exists for the same vague, same typeActivite, same dateDebut (WAT date), and statut = PLANIFIEE.
2. If found, skip creation.
3. This guards against the rare case where two concurrent processes (event-driven + CRON running simultaneously, EC-9.2) attempt to create the same activity.

### 3.7 Performance Considerations

- **Batch evaluation:** Process vagues in batches of 20. Each batch loads rules and recent releves in a single query using Prisma `include`.
- **Vague state caching:** After computing a vague's state (living count, average weight, FCR, etc.), cache it in a `vagueStateCache` Map within the request lifecycle. This avoids recalculating for each rule.
- **Rule pre-filtering:** Before evaluating conditions, filter rules by `typeDeclencheur` matching the current trigger source (CRON vs event).
- **Index usage:** `RuleExecution` has a composite unique index on `[regleId, vagueId, executionDate, triggerHash]` enabling O(1) idempotency checks.
- **Dead vague skip:** Skip vagues where `statut != EN_COURS` or where the associated site is inactive.

## 4. Pack Provisioning Architecture

### 4.1 Transactional Provisioning Flow

Pack activation creates 6+ entities atomically using `prisma.$transaction()`. If any step fails, the entire operation rolls back.

### 4.2 Entity Creation Order

```
prisma.$transaction(async (tx) => {
  // 1. Create or reuse client User
  const clientUser = await tx.user.upsert({
    where: { phone: clientPhone },
    create: { name, phone, passwordHash, role: PISCICULTEUR },
    update: {}
  });

  // 2. Create client Site
  const clientSite = await tx.site.create({
    data: { name: `Ferme ${clientUser.name}`, isActive: true }
  });

  // 3. Create SiteMember entries
  //    - Client as PISCICULTEUR on their site
  //    - DKFarm engineer as INGENIEUR on client site (read-only monitoring)
  //    - DKFarm admin as ADMIN on client site (support access)
  await tx.siteMember.createMany({ data: [...] });

  // 4. COPY ConfigElevage from Pack to client site
  const sourceConfig = await tx.configElevage.findUnique({
    where: { id: pack.configElevageId }
  });
  const clientConfig = await tx.configElevage.create({
    data: {
      ...sourceConfig,          // spread all fields
      id: undefined,            // new ID
      siteId: clientSite.id,    // client's site
      sourcePackId: pack.id,    // provenance tracking
      isDefault: true,          // default for this site
    }
  });

  // 5. COPY Produits from PackProduit to client site stock
  for (const pp of pack.produits) {
    const sourceProduit = await tx.produit.findUnique({ where: { id: pp.produitId } });
    const clientProduit = await tx.produit.create({
      data: {
        nom: sourceProduit.nom,
        categorie: sourceProduit.categorie,
        unite: sourceProduit.unite,
        uniteAchat: sourceProduit.uniteAchat,
        contenance: sourceProduit.contenance,
        prixUnitaire: sourceProduit.prixUnitaire,
        stockActuel: pp.quantite,       // initial stock from pack
        seuilAlerte: sourceProduit.seuilAlerte,
        fournisseurId: null,            // no supplier link on client site
        siteId: clientSite.id,
      }
    });
    // Create initial stock movement
    await tx.mouvementStock.create({
      data: {
        produitId: clientProduit.id,
        type: ENTREE,
        quantite: pp.quantite,
        date: new Date(),
        userId: adminUserId,    // DKFarm admin performing the activation
        siteId: clientSite.id,
        notes: `Stock initial - Pack ${pack.nom}`,
      }
    });
  }

  // 6. Create Vague
  const vague = await tx.vague.create({
    data: {
      code: generateVagueCode(),    // V-YYYY-NNN
      dateDebut: new Date(),
      nombreInitial: pack.nombreAlevins,
      poidsMoyenInitial: 7.0,       // default fingerling weight
      origineAlevins: `Pack ${pack.nom}`,
      statut: EN_COURS,
      siteId: clientSite.id,
      configElevageId: clientConfig.id,
    }
  });

  // 7. Create default Bac
  const bac = await tx.bac.create({
    data: {
      nom: "Bac principal",
      volume: 1.0,                  // placeholder, client updates later
      nombrePoissons: pack.nombreAlevins,
      vagueId: vague.id,
      siteId: clientSite.id,
    }
  });

  // 8. COPY RegleActivite rules (isGlobal=true) to client site
  const globalRules = await tx.regleActivite.findMany({
    where: { siteId: dkfarmSiteId, isGlobal: true, isActive: true }
  });
  await tx.regleActivite.createMany({
    data: globalRules.map(r => ({
      ...r,
      id: undefined,
      siteId: clientSite.id,
      isGlobal: false,          // client copy is not global
    }))
  });

  // 9. Create PackActivation
  const activation = await tx.packActivation.create({
    data: {
      code: generateActivationCode(),   // ACT-YYYY-NNNNN
      packId: pack.id,
      clientUserId: clientUser.id,
      clientSiteId: clientSite.id,
      vagueId: vague.id,
      configElevageId: clientConfig.id,
      statut: ACTIVE,
      siteId: dkfarmSiteId,            // seller site
      dateExpiration: addMonths(new Date(), 12),
    }
  });

  // 10. Trigger initial activity generation for week 1
  await generateActivitiesForVague(tx, vague, clientConfig, clientSite.id);

  return activation;
});
```

**Resolution of F-14:** Products are COPIED to the client site with `fournisseurId: null`. The client's stock is independent of DKFarm's inventory.

**Resolution of F-04:** The Pack itself remains on the DKFarm site (`Pack.siteId = dkfarmSiteId`). The client never queries Pack directly. The client interacts with their own Site's Vague, Activites, ConfigElevage, and Produits.

### 4.3 Code Generation

```typescript
// Pack codes: PACK-YYYY-NNN
async function generatePackCode(tx: PrismaTransaction, siteId: string): Promise<string> {
  const year = new Date().getFullYear();
  const prefix = `PACK-${year}-`;
  const last = await tx.pack.findFirst({
    where: { code: { startsWith: prefix } },
    orderBy: { code: 'desc' },
  });
  const seq = last ? parseInt(last.code.slice(-3)) + 1 : 1;
  return `${prefix}${String(seq).padStart(3, '0')}`;
}

// Activation codes: ACT-YYYY-NNNNN
async function generateActivationCode(tx: PrismaTransaction): Promise<string> {
  const year = new Date().getFullYear();
  const prefix = `ACT-${year}-`;
  const last = await tx.packActivation.findFirst({
    where: { code: { startsWith: prefix } },
    orderBy: { code: 'desc' },
  });
  const seq = last ? parseInt(last.code.slice(-5)) + 1 : 1;
  return `${prefix}${String(seq).padStart(5, '0')}`;
}
```

### 4.4 Error Handling and Rollback

The entire provisioning runs inside `prisma.$transaction()`. Any thrown error causes a full rollback. The API returns a descriptive error:

- **User creation conflict:** "Un utilisateur avec ce numero de telephone existe deja" (409)
- **Pack inactive:** "Ce pack n'est plus disponible" (400)
- **ConfigElevage missing:** "Aucune configuration d'elevage associee a ce pack" (400)
- **Code generation overflow:** Should never happen with 5-digit codes (99,999/year), but if it does, return 500 with a clear message

## 10. Technical Decisions

### 10.1 CRON: Vercel Cron Jobs

**Decision:** Use Vercel Cron Jobs for the daily activity generation.

**Rationale:**
- Native integration with the deployment platform (no additional service)
- Free tier supports daily execution (sufficient for MVP)
- The CRON endpoint is a standard API route, easily testable locally via `curl`
- If Vercel is later replaced, the endpoint remains a standard HTTP API callable by any scheduler

**Fallback:** If Vercel Cron proves insufficient (e.g., timeout issues with many clients), migrate to Upstash QStacks or a dedicated worker using the same API endpoint.

### 10.2 Template Engine: Simple String Replacement

**Decision:** Use `String.replace()` with `{placeholder}` syntax. No Handlebars or similar library.

**Rationale:**
- The placeholder vocabulary is finite and controlled (see Section 3.5)
- No conditional logic needed in templates (conditions are in the rule evaluation logic)
- Zero additional dependencies
- Easy for admin users to understand when editing rule templates

**Implementation:**
```typescript
function resolveTemplate(template: string, context: Record<string, string | number>): string {
  return template.replace(/\{(\w+)\}/g, (match, key) => {
    const value = context[key];
    return value !== undefined ? String(value) : '[non disponible]';
  });
}
```

### 10.3 ConfigElevage JSON Validation: Zod at API Layer

**Decision:** Validate `alimentTailleConfig` and `alimentTauxConfig` JSON fields using Zod schemas in the API route handlers.

**Schemas:**

```typescript
const AlimentTailleEntrySchema = z.object({
  poidsMin: z.number().min(0),
  poidsMax: z.number().min(0),
  tailleGranule: z.string().min(1),
  description: z.string().min(1),
  proteines: z.number().min(0).max(100).optional(),
});

const AlimentTailleConfigSchema = z.array(AlimentTailleEntrySchema)
  .min(1)
  .refine(
    (entries) => {
      // Validate no gaps/overlaps in weight ranges
      const sorted = [...entries].sort((a, b) => a.poidsMin - b.poidsMin);
      for (let i = 1; i < sorted.length; i++) {
        if (sorted[i].poidsMin !== sorted[i - 1].poidsMax) return false;
      }
      return true;
    },
    { message: "Weight ranges must be contiguous with no gaps or overlaps" }
  );

const AlimentTauxEntrySchema = z.object({
  phase: z.nativeEnum(PhaseElevage),
  tauxMin: z.number().min(0).max(100),
  tauxMax: z.number().min(0).max(100),
  frequence: z.number().int().min(1).max(10),
  notes: z.string().optional(),
});

const AlimentTauxConfigSchema = z.array(AlimentTauxEntrySchema)
  .min(1)
  .refine(
    (entries) => entries.every(e => e.tauxMin <= e.tauxMax),
    { message: "tauxMin must be <= tauxMax for every entry" }
  );
```

**Additional validation rules enforced at the API layer:**
- Phase thresholds must be monotonically increasing: `seuilAcclimatation < seuilCroissanceDebut < seuilJuvenile < seuilGrossissement < seuilFinition` (EC-5.7)
- Benchmark thresholds must be properly ordered: `fcrExcellentMax < fcrBonMax < fcrAcceptableMax` (EC-5.6)
- `poidsObjectif > 0`, `dureeEstimeeCycle > 0`, `tauxSurvieObjectif` in [0, 100]
- `nombreAlevins >= 1` on Pack (EC-1.1)
- `quantite > 0` on PackProduit (EC-1.3)

### 10.4 Timezone: WAT (UTC+1)

**Decision:** All "day" calculations (day-since-start, daily mortality rate, CRON day boundaries, rate limit resets) use WAT (UTC+1).

**Implementation:**
```typescript
function getWATDate(date: Date = new Date()): Date {
  // Convert UTC to WAT (UTC+1)
  const watOffset = 1 * 60; // minutes
  const utcMs = date.getTime() + date.getTimezoneOffset() * 60000;
  return new Date(utcMs + watOffset * 60000);
}

function getWATDateString(date: Date = new Date()): string {
  return getWATDate(date).toISOString().split('T')[0];
}

function jourDepuisDebut(dateDebutVague: Date): number {
  const start = getWATDateString(dateDebutVague);
  const now = getWATDateString();
  return Math.floor((new Date(now).getTime() - new Date(start).getTime()) / 86400000);
}
```

**Rationale (EC-11.1, EC-11.2, F-16):** Cameroon operates entirely in WAT. Using UTC would cause day boundaries to fall at 23:00 local time, leading to off-by-one errors in "day since start" calculations and causing daily activities to be generated too early.
