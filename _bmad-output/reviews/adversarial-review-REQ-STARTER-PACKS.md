# Adversarial Review: REQ-STARTER-PACKS.md

**Date:** 2026-03-15
**Reviewer:** BMad Master (Adversarial Review Task)
**Document:** docs/requirements/REQ-STARTER-PACKS.md v1.1

## Findings

### F-01: BLOCKER — `Activite` model referenced as "existing" but proposed fields conflict with actual schema

Section 4.4 says "Le modele `Activite` existant sera enrichi" and proposes adding `regleId`, `instructionsDetaillees`, `conseilIA`, `produitRecommande`, `quantiteRecommandee`, `priorite`, `isAutoGenerated`, `phaseElevage`.

Issues:
- Existing `recurrence` field (enum `Recurrence`) conflicts with `RegleActivite.intervalleJours` — two competing recurrence mechanisms
- Existing `userId` is NOT NULL — who is the "user" for auto-generated activities? No system user concept
- `produitRecommande` is typed as `String?` (FK to Produit) with no `@relation` — dangling string with zero referential integrity

### F-02: HIGH — No `TypeActivite` values for several proposed rule types

Existing `TypeActivite` enum: `ALIMENTATION, BIOMETRIE, QUALITE_EAU, COMPTAGE, NETTOYAGE, TRAITEMENT, RECOLTE, AUTRE`. The rule catalog maps "Tri obligatoire" to `AUTRE` as a workaround. Overloading `AUTRE` makes filtering and reporting meaningless. Document does not propose extending `TypeActivite`.

### F-03: BLOCKER — No userId for auto-generated activities on client site

Provisioning flow (section 3.1) says "Activites de la Semaine 1 auto-generees" on the client site. `Activite` requires mandatory `userId`. During auto-provisioning, who is userId? If DKFarm admin, it's a multi-tenancy data leak. If client, they didn't create them.

### F-04: HIGH — `Pack.siteId` creates conceptual collision with R8

Pack model has `siteId` pointing to DKFarm (seller) site. Client queries `WHERE siteId = mysite` return nothing. Document never specifies how client accesses pack information from their own site context.

### F-05: HIGH — `PackActivation.clientSiteId` is `@unique` — only ONE activation per site, ever

A client site can only ever have ONE pack activation. Second pack purchase cannot activate on same site. Contradicts "recurring revenue" business model.

### F-06: HIGH — `PackActivation.vagueId` is `@unique` — only ONE vague per activation

If client does "tri" (sorting) and splits fish into multiple vagues/bacs (section 4.5.3 explicitly instructs this), the one-vague constraint breaks.

### F-07: HIGH — No `configElevageId` on Vague in proposed schema

Section 10.1 says Vague gets `+ configElevageId?` but section 6.2 (ConfigElevage model) never shows relation FROM Vague TO ConfigElevage. Section 6.13 adds `configElevageId` to Pack, not Vague. Contradictory.

### F-08: HIGH — ConfigElevage is a god-object with 45+ fields and unvalidated JSON blobs

~50 scalar fields + 2 JSON columns (`alimentTailleConfig`, `alimentTauxConfig`). No JSON schema validation at DB level. Malformed JSON silently corrupts the activity engine.

### F-09: MEDIUM-HIGH — NoteIngenieur queries need `clientSiteId` but no client endpoint exists

No `/api/notes` endpoint for CLIENT to read engineer notes. Client cannot see advice unless they access engineer-only pages.

### F-10: MEDIUM-HIGH — No permission model for Phase 3 features

Zero new permissions proposed for Packs, ConfigElevage, ConseilIA, NoteIngenieur, RegleActivite, Engineer dashboard. No SUPER_ADMIN or INGENIEUR role exists in the `Role` enum.

### F-11: MEDIUM-HIGH — Activity Engine execution model is hand-waved

- Next.js has no built-in CRON — mechanism never specified
- No idempotency mechanism — duplicate daily activities if CRON runs twice
- No performance budget for per-releve rule evaluation

### F-12: MEDIUM — AI rate limiting mentions "pack standard" tier but no tier field exists

Pack model has no `aiQuota` or tier field. No data to enforce tiered rate limiting.

### F-13: MEDIUM — ConseilIA model has no `siteId` relation defined

Shows `siteId String // R8` but no `site Site @relation(...)`. Same for `activiteId`.

### F-14: MEDIUM — Pack activation must COPY Produit records to client site

PackProduit references `produitId` on DKFarm site. Client needs Produit records on THEIR site. Copy logic never specified. What about `fournisseurId` on copied Produit?

### F-15: MEDIUM — `RegleActivite.siteId` makes rules site-specific, but seed is "global"

If rules are per-site, every activation must COPY all seed rules. Never mentioned. If rules are global, `siteId` violates R8.

### F-16: LOW-MEDIUM — Timezone handling for "jour depuis debut" calculations

Cameroon is UTC+1. `DateTime` stores UTC. Day calculations off by one depending on start time.

### F-17: MEDIUM — Notifications push (PWA/SMS) with zero architecture

Multi-week integrations presented as casual bullet points. No sprint allocation, API design, or cost estimation.

### F-18: LOW-MEDIUM — "Pack Custom" has no dynamic calculation support

Indistinguishable from regular pack once saved. UI flow never mentions how custom calculation works.

### F-19: LOW-MEDIUM — Template placeholders have no formal specification

No list of available placeholders, no template engine spec, no error handling for unresolvable placeholders.

### F-20: LOW — Metrics of success have no implementation path

"Taux de reachat" and "Satisfaction utilisateur" — neither tracking mechanism nor feedback system specified.

### F-21: LOW — Benchmarks par phase explicitly punted to "v2"

Half-designed feature will confuse implementers. Either spec fully or remove.

### F-22: LOW — ConfigElevage `isDefault` has no uniqueness constraint

No `@@unique` preventing multiple default configs per site.

### F-23: LOW — No time estimates for 6 sprints of Phase 3

No story points, sprint duration, team capacity, or risk assessment.
